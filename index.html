Commonly Used Terms

Here are some of the commonly used terms/names at O'Reilly:

FTP Date
Files to Printer Date. The date that you send the completed book to the printer. The "last FTP date" is the last date of the month that you can send a book to the printer and have it released that same month.
OTD
Out the Door. A final review performed by a senior member of the production staff. Usually done the day before the FTP date. About a week before the FTP date, send a request to otd@oreilly.com asking for someone to do the OTD.
Touchstone
Touchstone is the shared server at O'Reilly, and is accessed via the command line. You will need to log onto Touchstone to perform some functions.
SVN
Subversion. This is a version control system that is used to hold covers, PDFs, and a few other things. This will mostly be used during Post-Production.
Intake
Intake is process during which the Tools team takes the content submitted by the editor and makes it production ready. This is generally the first phase of production, and is started automatically once a book arrives in Production. Once it's done, Tools will send an email with any relevant notes.
Conversion
Conversion is the process in which books written in MS Word are converted into HTML and imported into Atlas. Unlike Intake, this is usually done after the Copyedit phase.
Vendor
A vendor is a freelancer who works on a book. We generally use vendors for copyediting, proofreading, and indexing. Occasionally a vendor will help with entering edits. Vendors are hired by Melanie Yarbrough. Production Editors are responsible for requesting vendors to help with their project.
1C/4C
1 Color / 4 Color. This refers to the number of inks used in the printing process. 1 Color is grayscale and is used in the interior for the majority of O'Reilly books. 4 Color is also known as CMYK and is used in the interior of select O'Reilly books, as well as most Make: books. Production Editors will have to run PitStop Profiles to convert PDFs to the correct color profile before sending the final files to the printer.
Repo
Short for repository. Basically, it's the folder containing all the files that make up the book. Repos are kept in Atlas, and can be found at the Atlas URLs.
PDB
Product Database. Where all the information about a book is stored.

Markup Tips

Markup language is code used to translate raw text into a formatted book. When a book gets to production and finishes Intake it will be fully marked up. Prod Eds then edit markup as needed during the various review periods. There are two main ways to edit the markup: Atlas or Locally. Editing in Atlas is fast and direct, while editing locally (using Git) allows a little more customization and control. You can do either or both, as you prefer.

Entities

Entities are HTML codes that translate into special characters. They are useful for inserting characters that are not easily written on a normal keyboard, or in cases where you want to insert a character that would otherwise be interpreted as markup. Several commonly used entities are list below:

  • Em Dash (—): &#x2014;
  • En Dash (–): &#x2013;
  • Greater Than: (>): &#x3e;
  • Less Than: (<): &#x3c;
  • Ampersand: (&): &#x26;

Passthroughs

Passthroughs are used in asciidoc markup to have a section use HTML markup. This is often done because HTML tends to be more precise than asciidoc, making it necessary to render complex areas correctly. There are two types of passthroughs inline and block.

Inline
Used for small sections. The passthrough is written like so: pass:[<tag>passthrough section</tag>].
Example: "...and we'll return the pass:[<code>TaskInfo</code>]s that will launch those pass:[<code>Job</code>]s."
Block
Used for larger sections that span multiple lines. Passthrough sections are written as four plus signs (+) above and below the section.
Example:
++++
<pre data-type="programlisting">
# <strong>zbid</strong>
Dev Product String Serial Number
002:005 KILLERB001 [DELETED]
</pre>
++++

Git

Git is the version control system used by Atlas. Basically, Git allows users to work locally (i.e. your computer) and then merge their changes back to the central repository (i.e., Atlas). You can use Git via the command line (i.e. the Terminal app on Mac). This is useful at times when you may not have internet access, or if you prefer to use a text editor software rather than the browser-based Atlas code editor.

Installing Git

Before you can use Git, it has to be installed. Go to the official Git site, and follow the instructions.

Using Git

Git can be used for a variety of things, but luckily we only need to use a few simple commands (listed below).

git clone
This "clones" (i.e., makes a copy of) the repo onto your computer. Once a repo is cloned, you can make changes to the files in the repo locally. This will usually be the first command you run. To use type: "git clone Git URL". You can find the Git URL on the Project Settings page for an Atlas projects.
Example:
git clone git@git.atlas.oreilly.com:oreillymedia/client-side-data-storage.git
git pull
This "pulls" any changes from the central repo (Atlas), and incorporates them into your local files. You should do this before making any local changes, to make sure your local repo is up to date.
git commit
This finalizes any changes you've made locally, and readies them to be pushed to the central repo
This takes some additional arguments, you will generally use "-a" and "-m". -a means that you are committing changes made in all the files in the repo. -m means that you are including a message, and is followed by a message in quotes. The message should indicate what the commit did.
Example:
git commit -a -m "Entering edits into Chapter 1, 2, and 3"
git push
This "pushes" any committed changes to the central repo. Basically, any changes you made locally are added into the Atlas repo
git add
Similar to git commit, but adding an entirely new file to the repo, rather than changing a file.

The Product Database

The Product Database (PDB) contains a great deal of information about every book, and you will likely refer to it often. It can be a little daunting at first, but once you figure out where things are it's not too bad.

The PDB is split into several tabs based on departments (Editorial, Production, Design, Manufacturing, and Marketing).

There are usually several listings for a book based on format, but the most commonly used one is the "Tech Book" listing, which is the print book.

Where to Find Things in the PDB

Final Page Count: Producion tab. Prod Eds enter this number when the book is nearly complete. Calculated by adding the interior PDF plus any backmatter ads and blanks (see Printer Pagecount Specifications).

Back Cover Copy: Design tab → Cover Req sub tab. Ed Stephenson will send this to the Prod Ed, who will enter this in to the PDB. The design team will reference this when creating the final cover.

Retail Availabity Date: Manufacturing tab. You need this date in Post-Production, and sometimes authors will ask when their book will be available. This date is based on the FTP date, which Prod Eds schedule with Manufacturing.

Consumer Short Text: Marketing tab. This is needed for Post-Prod. It's basically a short summary for marketing purposes

Shelving Category: Production tab. This is an important element on the back cover, and should be checked when reviewing draft cover mechs.
